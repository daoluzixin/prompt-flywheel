# Prompt 优化数据飞轮：实践策略手册

> 来源：30 轮 GAN 风格迭代实验（7 KEEP / 4 REJECT + 飞轮闭环 6 轮 + L3 泛化验证 + 稳定性验证 5 轮 + 独立 Pass 架构 5 轮）
> 适用范围：LLM 信息提取 / 分类 / 结构化输出场景的 prompt 优化
> 最终成绩：Dual-Pass v2.1 — PI F1=0.84, KG F1=0.87；约 80 行 prompt 增量 + GT 46→81 条标注 + 架构拆分为独立 Pass

---

## 方法论总述

本套优化采用 GAN 风格的迭代框架——prompt 作为 Generator，评估系统（eval-runner + 冻结 Ground Truth）作为 Discriminator。每轮只改一个变量，跑完整 eval 看指标涨落，达标则 KEEP、退化则 REJECT 并回滚。核心假设是 prompt 的改进空间可以通过 bad case 驱动的局部修复逐步逼近全局最优，但存在架构级天花板。

这个假设在实验中得到了完美验证：前 9 轮稳步攀升；R10-R11 触及 prompt 层面天花板（跨维度耦合硬约束）；飞轮阶段通过 GT 补完突破该天花板；R18 的 L3 泛化探针验证了 prompt 的真正泛化能力；R26-R30 通过架构拆分彻底消除了跨维度耦合，并在独立 pass 上实现了 KG F1 从 0.63→0.87 的历史最大跃进——证明架构拆分后单维度 prompt 的优化空间远大于耦合状态。

---

## 有效策略

**单一目标 + 单变量改动**是所有策略的前提。7 个 KEEP 轮次无一例外每轮只攻一个失败模式、改动控制在 5-25 行内。这保证了精确归因——改了三处同时涨了你不知道哪处生效，退化了也不知道该回滚哪处。Round 4 只增加 spec3.plan 段落（25 行），recall 从 40%→70%，其余 category 不受影响，就是典型的单变量成功案例。

**正例教学**解决"模型不知道应该识别"的问题。当某 category recall 持续极低时，根因往往是 prompt 中压根没教过这种模式。解法是补充反直觉的边界案例作为正例——选那些模型本来不会识别的样本（极短文本、非典型格式），而非模型已经能处理的典型样本。Round 1 为 spec1.spy 补充 3 个短批注正例，recall 从 0.21 直接跳到 1.00。这类问题的天花板几乎不存在，只要正例选得准，recall 提升立竿见影。

**精准反例**解决"模型过度生成"的问题。反例要具体到能直接匹配实际 FP 的语义模式（如"spec 流程规范""模板设计说明"），不能是抽象的排除条件。Round 2 为系统架构列出 5 类 FP 模式后，Precision 从 0.62 跳到 0.90，Recall 不降反升。但这个策略存在天花板：当 FP 与 TP 语义高度耦合时（来自同一文档的同类文本），反例会同时压低 TP——Round 11 验证了这一点（FP 归零但 TP 从 6 降到 3）。

**一句话经验法则**比形式化定义更有效。一个简单的二分测试让模型在模糊场景中快速判断。Round 3 用"what to do→spec4；how to do→spec3/spec5"一句话解决了 spec4 从 recall 0.00→1.00，此前三段正式定义毫无效果。R30 的 3 条 category 区分规则让业务规则和接口约束双双达到 F1=1.00，ROI 极高。

**"建议性规则"升级为"带执行步骤的强制检查"**能带来稳定提升。告诉模型"注意检查 X"效果有限，改为明确的执行步骤（"输出 JSON 前，逐条回顾每条 PI，检查上下文是否包含文件命名规则/维度变更/通信方式等伴生知识点"）效果显著。Round 9 将伴生知识规则从"建议"升级为"强制二次检查"，KG Recall 从 0.67 飙升至 0.90（+24pp）。LLM 对可操作的步骤序列的遵从度远高于抽象建议。

**用"独立性原则"替代"合并原则"**。对 LLM 说"相似的条目应合并"会严重抑制输出——模型会泛化为"能不输出就不输出"。Round 5a 添加"合并原则"后 recall 暴跌 12pp。正确做法是反转表述：强调"每条独立事实性错误单独计数"，用正向表述维持 recall。Round 5b 做了这个反转后 PI F1 首次破 0.84。

**维度间规则严格隔离**。多维度提取任务中各维度的判定规则必须各自独立，不能交叉引用。任何跨维度的提及——无论是"此规则不影响另一维度"还是"这不是 X 而是 Y"——都会被模型理解为不确定性信号，对两边都变保守。Round 7 和 Round 10 两次都因跨维度引用导致全面退化。

**合成对抗探针验证泛化边界**。当 prompt 在已知数据集上趋近满分、剩余 FN 难以归因时，基于已知 failure patterns 构造措辞/场景变体的对抗样本是确认答案的有效手段。R18 设计了 10 个 synthetic probes，prompt v1.9 取得 10/10 检出率，证明 R17 的 3 个 FN 是 subagent 随机波动而非 prompt 盲区。探针设计的关键是变体要足够远离已知 case 的原始措辞——用"不对"测试极短否定识别，用"能不能简洁点"测试提问式纠错泛化。

## 已验证的反模式

4 次 REJECT 反复验证了同一个核心洞察：LLM 对限制性指令的敏感度远超预期。

**限制性语义表述**必须避免。不要对 LLM 说"合并""减少""去重""精简"——任何减量语义都会全局性抑制输出，即使加了严格的作用范围限定。替代方案是用正向表述强调独立性和完整性。

**排除规则的元声明**会适得其反。不要在排除规则旁添加"另一维度不受影响""此规则仅限 X"等元声明——模型会把跨维度提及理解为鼓励多输出的信号，导致被排除的维度反而出现更多 FP。让每个维度的规则完全自包含，对另一维度保持沉默。

**跨维度宏观边界规则**会让两个维度同时退化。不要写"如果内容是 KG 则不要输出为 PI"这类规则——模型会对规则中提到的每个维度都产生不确定性。每个维度独立描述"什么属于本维度"即可。

**"不要输出 X"形式的反例**在高 recall category 上会压制 TP。当 FP 与 TP 语义耦合时，无论反例多么精确（即使引用具体 quote），模型都会泛化为对整个 category 保守。Round 11 中 3 条精确反例让 FP 归零的同时 TP 从 6 暴跌至 3。替代方案是接受少量 FP 作为高 recall 的正常代价，或寻求架构级解法。

## 工程纪律

**Baseline 方差是判断基准。** 用同一 prompt 跑多轮建立方差基线。本实验 4 轮早期基线 KG σ=0.12；R19-R23 的 5 轮稳定性验证给出了更精确的值 σ=0.08。这意味着后续任何优化增益必须超过 0.10 F1 才能被可靠判定为真实改进。不建立方差就无法判断改动是否真正有效。方差基线还能区分"单轮离群点"和"系统性退化"——R22 的 PI F1=0.63 低于均值 1.5σ，但 R23 立即恢复高位，结合根因定位可确认为随机波动。

**GT 质量决定优化天花板。** 在怀疑 prompt 达到极限前先审查 Ground Truth。Round 8 发现 2 条 PI 的文档归属标注错误，修正后 PI F1 单轮涨 7pp——这比任何 prompt 改动的效果都大。飞轮阶段系统性发现了 5 条 GT 遗漏，补入后 F1 从 0.91 提升到 0.94，且无需改动任何 prompt 行。

**回滚机制是安全网。** 每次 KEEP 后立即备份当前版本，REJECT 时一条 cp 命令即可回滚。4 次 REJECT 都靠 backup 零成本恢复。

**GT 与 prompt 共同进化优于单独优化任一方。** 这是飞轮区别于 DSPy/OPRO 等固定数据集框架的核心差异。当 prompt 改进使模型开始识别 GT 中未标注的真实信号时，这些"FP"实际上是 GT 遗漏——补入后 FP 自然消除、F1 自然提升。数据和 prompt 的共同进化形成正反馈循环，这就是"飞轮"而非"迭代"的含义。

## 架构级突破

当数据飞轮连续多轮 REJECT 且所有 prompt 层策略均已尝试时，说明已触及当前架构的天花板。

**独立 Pass 拆分**是本实验验证的最重要架构决策。将多维度提取拆为独立的两次调用，每次只关注一个维度，从根本上消除跨维度语义耦合。代价是 token 成本翻倍，但收益远超代价：PI F1 从 0.75→0.84（spec5.impl FP 从 5.8→1.3），且拆分后的单维度 prompt 获得了独立优化的自由度。R29-R30 在 KG-only pass 上一次性将 KG F1 从 0.63 提升到 0.87——在耦合状态下这种改动根本不可能。

**"冻结一侧、迭代另一侧"**是架构拆分后的巨大优势。冻结已稳定的 PI pass，只迭代 KG pass，eval 成本降低 50%，且彻底消除了"改 KG 影响 PI"的风险。

**"PRD 输入 vs 纠正性补充"区分规则**解决了 FP 来自需求文档原文的问题。关键区分是"这条信息是用户在 AI 犯错后才补充的纠正性知识，还是需求文档本身就包含的业务描述"。R29 通过一条明确的区分规则将此类 FP 从 8 条直接降为 0，Precision +18pp。

**L3 泛化探针作为收敛判据**。当 prompt 在已知数据上稳定满分但剩余 FN 难以归因时，对抗探针提供了一种客观的收敛判据：如果 prompt 在所有已知 failure patterns 的措辞变体上仍能正确识别，则剩余 FN 可归因于 LLM 采样噪声，优化可暂停。这比"连续 N 轮无变化就停"更有信息量——它区分了"没有改进空间"和"改进空间存在但当前策略不够"。

**稳定性验证作为进入下一阶段的前提**。多轮重复评估的价值不仅是"确认稳定"，更重要的是为后续优化提供精确的显著性判据。没有这个数字，任何改动都无法判断是真实改进还是采样噪声。同时它还能暴露"系统性弱点"与"随机波动"的区别，直接决定下一步的优化方向——攻系统性弱点而非试图"修复"随机波动。

## 速查决策表

| 观察到的现象 | 根因判断 | 推荐策略 | 预期效果 |
|-------------|---------|---------|---------|
| 某 category recall 持续极低 | 模型不知道该识别此模式 | 补充边界正例（3-5 个） | recall 大幅提升 |
| 某 category FP 过高 | 模型不知道边界在哪里 | 列出"什么不是 X"的反例 | precision 大幅提升 |
| TP 被归错相邻 category | 缺乏类别间区分规则 | 一句话经验法则 | 混淆率归零 |
| 规则为"建议性"时效果不稳定 | 模型有时遵循有时忽略 | 升级为带步骤的强制检查 | recall 稳定提升 |
| 添加"合并/去重"后 recall 暴跌 | 限制性语义被泛化 | 反转为"独立性原则" | recall 恢复 |
| 修 FP 时 TP 同步下降 | FP 与 TP 语义耦合 | 接受 FP 或拆 pass | — |
| 连续 2+ 轮 REJECT | 触及 prompt 天花板 | 转向架构级方案 | — |
| 剩余 FN 难以归因 | 泛化能力未验证 | 构造对抗探针 | 确认是否为系统性盲区 |
| 单轮指标异常低 | 缺乏稳定性基线 | 多轮重复 + 根因定位 | 离群点归因 |

## 适用条件与局限

本手册的策略基于以下实验条件：任务为信息提取 + 分类（非生成式创作），评估为精确匹配（按 contentId + category 计数），GT 规模为 28 篇 golden 文档 / 81 条标注，模型为单次调用，6 个 subagent 并行。对于生成式任务、开放域问答、或评估标准为人类偏好评分的场景，精准反例和经验法则可能需要适配。但"单变量改动""限制性语义敏感""维度隔离"等核心原则具有普适性——它们反映的是 LLM 对指令的理解机制，而非特定任务的特性。
