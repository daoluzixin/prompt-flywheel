# Prompt 优化数据飞轮：实践策略手册

> 来源：30 轮 GAN 风格迭代实验（prompt 迭代 7 KEEP / 4 REJECT + 飞轮闭环 6 轮 + L3 泛化验证 1 轮 + 稳定性验证 5 轮 + 独立 Pass 架构 3 轮 + KG 精调 2 轮）
> 适用范围：LLM 信息提取 / 分类 / 结构化输出场景的 prompt 优化
> 最终成绩：独立双 Pass 架构 v2.1 — PI F1=0.84, KG F1=0.87, Combined F1≈0.85；spec5.impl FP=1（vs 单 Pass 基线 5.8）
> 约 80 行 prompt 增量改动 + GT 从 v3.0→v3.4（46→81 条标注）+ 架构拆分为 PI-only + KG-only 独立 Pass

---

## 一、总体方法论

本套优化采用 GAN 风格的迭代框架——G（analyze-batch-prompt）作为被优化对象，D（eval-runner + 冻结 Ground Truth）作为判别器。每轮只改一个变量，跑完整 eval 看指标涨落，达标则 KEEP、退化则 REJECT 并回滚。核心假设是 prompt 的改进空间可以通过 bad case 驱动的局部修复逐步逼近全局最优，但存在架构级天花板——前 9 轮稳步攀升，R10-R11 触及 prompt 层面天花板（spec1.spy FP=3 的跨维度耦合硬约束）；飞轮阶段（R12-R17）通过 GT 补完突破了该天花板（确认部分 FP 实为 GT 遗漏）；R18 的 L3 泛化探针进一步验证了 prompt 的真正泛化能力而非对已知 case 的过拟合。R26-R28 验证了独立 Pass 架构的可行性（消除跨维度耦合），R29-R30 在 KG-only pass 上实现了 KG F1 从 0.63→0.87 的历史最大跃进——证明架构拆分后单维度 prompt 的优化空间远大于耦合状态。

## 二、八条已验证的有效策略

**单一目标 + 单变量改动。** 所有 7 个 KEEP 轮次的共性是每轮只攻一个失败模式、改动控制在 5-25 行内。这保证了精确归因——如果改了三处同时涨了你不知道哪处生效，退化了也不知道该回滚哪处。Round 4 只增加 spec3.plan 段落（25 行），spec3.plan recall 从 40%→70%（+30pp），其余 category 不受影响，就是典型的单变量成功案例。

**正例教学解决"不知道应该识别"。** 当模型系统性漏检某类输出时，根因往往是 prompt 中压根没教过这种模式——模型不是"不愿识别"而是"不知道应该识别"。解法是补充反直觉的边界案例作为正例：选那些模型本来不会识别的样本（极短文本、非典型格式），而非模型已经能处理的典型样本。Round 1 为 spec1.spy 补充 3 个短批注正例（"MQ消息""支持可配置""这些不全"），recall 从 0.21 直接跳到 1.00。

**精准反例解决"过度生成"。** 当某 category FP 过高时，列出"什么不是 X"的具体反例效果极佳——反例要具体到能直接匹配实际 FP 的语义模式（如"spec 流程规范""模板设计说明""框架愿景"），不能是抽象的排除条件。Round 2 为系统架构列出 5 类 FP 模式后，KG Precision 从 0.62 跳到 0.90（+45%），且 Recall 不降反升。但这个策略存在天花板：当 FP 与高 recall 的 TP 语义高度耦合时（如 spec1.spy 的 FP 和 TP 都来自同一篇文档的短批注），反例会同时压低 TP——Round 11 验证了这一点。

**经验法则比形式化定义更有效。** 一句话的判断规则比三段落的定义描述更能帮模型做分类决策。经验法则应提供一个简单的二分测试，让模型在模糊场景中快速判断。Round 3 用"what to do（做什么）→spec4；how to do（如何设计/实现）→spec3/spec5"一句话解决了 spec4.tasks 从 recall 0.00→1.00——此前三段正式定义毫无效果。

**"建议性规则"升级为"带执行步骤的强制检查"。** 告诉模型"注意检查 X"效果有限——模型有时遵循有时忽略。改为明确的执行步骤（"输出 JSON 前，逐条回顾每条 PI，检查上下文是否包含文件命名规则/维度变更/通信方式等伴生知识点"）效果显著且稳定。LLM 对可操作的步骤序列的遵从度远高于抽象建议。Round 9 将伴生知识规则从"建议"升级为"强制二次检查"，KG Recall 从 0.67 飙升至 0.90（+24pp）。

**用"独立性原则"替代"合并原则"。** 对 LLM 说"相似的条目应合并"会严重抑制输出——即使限定了"只限完全相同"，模型也会泛化为"能不输出就不输出"。Round 5a 添加"合并原则"后 recall 暴跌 12pp，spec1.spy recall 从 1.00 降至 0.67。正确做法是反转表述：强调"每条独立事实性错误单独计数"，用鼓励输出的正向表述维持 recall。Round 5b 做了这个反转，PI F1 首次破 0.84。

**维度间规则严格隔离。** 多维度提取任务（如同时提取 PI 和 KG）中，各维度的判定规则必须各自独立，不能交叉引用。任何跨维度的提及——无论是"此规则不影响另一维度"还是"这不是 X 而是 Y"——都会被模型理解为不确定性信号，对两边都变保守。Round 7（元声明"此排除仅限 KG 维度"）和 Round 10（PI/KG 边界规则）两次都导致全面退化。

**合成对抗探针验证泛化边界。** 当 prompt 在已知数据集上趋近满分、剩余 FN 难以判断是系统性缺陷还是随机波动时，基于已知 failure patterns 构造措辞/场景变体的对抗样本（adversarial probes）是确认答案的有效手段。R18 基于 3 个 active failure patterns（短文档零输出、极短批注遗漏、提问式纠错遗漏）设计了 10 个 synthetic probes，prompt v1.9 在上面取得 10/10 检出率，证明 R17 的 3 个 FN 是 subagent 随机波动而非 prompt 盲区——这个结论避免了不必要的 prompt 改动（改了反而可能引入退化，如 R10/R11 的教训）。探针设计的关键是变体要足够远离已知 case 的原始措辞：syn-004 用"不对"（而非原始的 spec-mark 批注）测试极短否定识别，syn-007 用"能不能简洁点"（而非原始的"能不能更细一点"）测试提问式纠错泛化。

## 三、四类失败模式及最佳对策

十一轮实验揭示了四类本质不同的失败模式，每类有各自的最佳应对策略和天花板。

**系统性漏检（模型"不知道"）。** 表现为某 category recall 持续极低且跨轮次稳定（如 spec1.spy recall 0.21）。对策是正例教学 + 模式描述，选择模型无法自行推断的边界案例。这类问题的天花板几乎不存在——只要正例选得准，recall 提升立竿见影。

**过度生成（模型"太积极"）。** 表现为某 category FP 持续偏高（如系统架构 FP=7）。对策是精准反例 + 排除条件，列出具体的 FP 语义模式作为"不属于此类"的示例。天花板在于当 FP 与 TP 的语义高度耦合（来自同一文档、同一发言者、同一主题）时，任何形式的抑制都会同步降低 recall，此时需要架构级方案。

**Category 混淆（模型"分不清"）。** 表现为 TP 被归错类别（如 spec4 被误判为 spec3）。对策是边界经验法则 + 专属判定段落，用一句话的二分测试区分相邻类别。天花板是相邻阶段的 hard case，可能需要更细粒度的阶段定义。

**跨维度语义耦合（新发现）。** 表现为同一用户发言同时产出 PI 和 KG，在 prompt 中精确分割两者的归属时，无论用宏观规则还是精确反例都让模型对两个维度同时变保守。这类问题在 prompt 层面无有效解法——Round 10 和 Round 11 分别用完全不同的策略尝试均失败。需要架构级改变，如将 PI 和 KG 提取拆为两个独立 pass，或引入二阶段"提取→分类"pipeline。

## 四、四条已验证的反模式

4 次 REJECT 反复验证了同一个核心洞察：LLM 对限制性指令的敏感度远超预期。以下四种写法已被实验证伪，必须避免。

**限制性语义表述。** 不要对 LLM 说"合并""减少""去重""精简"。任何减量语义都会全局性抑制输出，即使加了严格的作用范围限定，模型也会泛化为保守策略。替代方案是用正向表述强调独立性和完整性——"每条独立事实性错误单独计数"比"相同错误不要重复输出"安全得多。

**排除规则的元声明。** 不要在排除规则旁添加"另一维度不受影响""此规则仅限 X"等元声明。模型会把这种跨维度提及理解为鼓励多输出的信号，导致被排除的维度反而出现更多 FP。替代方案是让每个维度的规则完全自包含，不引用其他维度的存在。

**跨维度宏观边界规则。** 不要写"如果内容是 KG 则不要输出为 PI"这类跨维度边界规则。模型会对规则中提到的每个维度都产生不确定性，结果是两个维度的输出都变少。替代方案是每个维度独立描述"什么属于本维度"，对另一维度保持沉默。

**"不要输出 X"形式的反例用于高 recall category。** 当一个 category 的 recall 很高但有少量 FP 时，不要用"不要输出以下场景"形式的反例来压制 FP——无论反例多么精确（甚至引用具体 quote），模型都会泛化为对整个 category 的输出保守。Round 11 中 3 条精确到原文级别的反例让 FP 归零的同时也让 TP 从 6 暴跌至 3。替代方案是接受少量 FP 作为高 recall 的正常代价，或寻求架构级解法。

## 五、工程实践要点

**Baseline 方差是判断基准。** 用同一 prompt 跑多轮建立方差基线。本实验中早期 4 轮基线（18-doc）KG F1 标准差为 0.12；R19-R23 的 5 轮稳定性验证（28-doc）给出了更精确的值：σ=0.08，这意味着后续任何优化的增益必须超过 0.10 F1 才能被可靠地判定为真实改进而非采样噪声。不建立方差就无法判断改动是否真正有效，容易把噪声当信号 KEEP 或把有效改动误判为退化 REJECT。方差基线还有一个实用价值：它能区分“单轮离群点”和“系统性退化”——R22 的 PI F1=0.63 低于均值 1.5σ 但 R23 立即恢复高位，结合根因定位（batch_1 的 spec1.spy subagent 单次采样偏差）可确认为随机波动而非 prompt 退化。

**GT 质量决定优化天花板。** 在怀疑 prompt 达到极限前先审查 Ground Truth。Round 8 发现 2 条 PI 的文档归属标注错误，修正后 PI F1 单轮涨 7pp——这比任何 prompt 改动的效果都大。GT 错误会制造假 FP/FN，误导优化方向，让你在错误的方向上耗费多轮迭代。

**回滚机制是安全网。** 每次 KEEP 后立即备份当前版本（`backup/analyze-batch-prompt-v{X}.md`），REJECT 时一条 cp 命令即可回滚。11 轮中 4 次 REJECT 都靠 backup 零成本恢复。没有备份的改动等于赌博——一旦退化且改了多处，你连回到上一个好版本都做不到。

**并行执行引入的方差不可消除。** 6 个 subagent 并行执行同一 prompt 时 batch 间结果有差异，这是 LLM 固有的随机性，不能通过 prompt 消除。应对方式是对关键决策（KEEP vs REJECT）设置 significance threshold——超过 baseline 方差才算显著，避免因噪声做出错误决策。R19-R23 进一步揭示了波动的结构性规律：Precision 的波动远小于 Recall（排除离群点后 PI Precision 仅在 0.75-0.76 间波动，Recall 则在 0.64-0.89 间大幅摆动），这说明 prompt 的“什么不该提取”规则已收敛，但“什么必须提取”仍受 subagent 注意力随机性影响——未来的优化应优先攻 Recall 稳定性而非 Precision。

**KEEP 与 REJECT 的共性规律。** 7 次 KEEP 的共性是单一目标、单变量改动、正向表述（教模型"什么是"而非"什么不是"）、不涉及跨维度引用。4 次 REJECT 的共性归为三类：(a) 间接限制性措辞（合并原则、元声明），(b) 跨维度宏观规则（PI/KG 边界），(c) 涉及"不要输出"的抑制性指导。这说明对 LLM prompt 的改动，正向教学（告诉它该做什么）远比负向约束（告诉它不该做什么）安全。

**GT 与 prompt 共同进化优于单独优化任一方。** 飞轮阶段（R12-R17）的核心发现是：当 prompt 改进使模型开始识别 GT 中未标注的真实信号时，这些"FP"实际上是 GT 遗漏——补入后 FP 自然消除、F1 自然提升。R15/R16 的 FP 分析系统性发现了 5 条 GT 遗漏，补入 v3.3 后综合 F1 从 0.91 提升到 0.94，且无需改动任何 prompt 行。同样，R18 将 10 个 synthetic probes 补入 GT v3.4 后，为后续迭代建立了更严格的回归基准——这意味着未来任何 prompt 改动不仅要通过原始 18 篇 golden 的测试，还要通过 10 个精心设计的对抗场景。数据和 prompt 的共同进化是飞轮区别于 DSPy/OPRO 等固定数据集框架的核心差异。

## 六、架构级突破方向（已验证）

当数据飞轮连续多轮 REJECT 且所有 prompt 层策略均已尝试时，说明已触及当前架构的天花板。本实验在 Round 10-11 明确触达了这一边界，并在 R26-R30 通过架构级改变成功突破。

**独立 Pass 拆分（✅ R26-R28 验证）。** 将多维度提取（如 PI + KG）拆为独立的两次调用，每次只关注一个维度，从根本上消除跨维度语义耦合。代价是 token 成本翻倍，但收益远超代价：PI F1 从 0.75→0.84（spec5.impl FP 从 5.8→1.3），且拆分后的单维度 prompt 获得了独立优化的自由度。R29-R30 在 KG-only pass 上一次性将 KG F1 从 0.63 提升到 0.87——在耦合状态下这种改动根本不可能（R10/R11 证明任何 KG 改动都影响 PI）。

**拆分后的"冻结一侧、迭代另一侧"策略（✅ R29-R30 验证）。** 架构拆分后的巨大优势是可以冻结已稳定的 pass（如 PI），只迭代需要改进的 pass（如 KG）。这将 eval 成本降低 50%（只跑 KG pass 的 6 个 subagent，复用 PI 结果），且彻底消除了"改 KG 影响 PI"的风险。R29 和 R30 均复用 R28 的 PI 结果，PI F1 严格不变（0.8378），KG 可以放心做激进优化。

**精细诊断驱动的 category boundary fix（✅ R30 验证）。** 当主要 FP/FN 来源于 category 误分类（如"订单号获取逻辑"被分到接口约束而非业务规则）时，通过精确的边界规则（一句话区分规则）可以直接纠正，且 ROI 极高。R30 仅添加 3 条 category 区分规则就让业务规则和接口约束双双达到 F1=1.00。

**"PRD 输入 vs 纠正性补充"区分规则（✅ R29 验证）。** 当 FP 主要来自模型将需求文档原文误判为知识库缺口时，关键区分是"这条信息是用户在 AI 犯错后才补充的纠正性知识，还是需求文档本身就包含的业务描述"。R29 通过一条明确的区分规则（加正反例）将此类 FP 从 8 条直接降为 0，Precision 提升 +18pp。

**二阶段 Pipeline。** 第一阶段宽松提取所有候选项（高 recall、低 precision），第二阶段对候选项逐条做精细分类和过滤。适合 FP 与 TP 语义高度耦合的场景——第一阶段不怕多提，第二阶段有充足上下文做精确判断。（尚未实验验证）

**GT 重新标注。** 当剩余 FP 中有明显的 GT 边界争议项时（如标注者对某条归属也会有分歧），应通过多人标注 + 一致性检验重新确认 GT，而非强行让 prompt 适配有争议的标注。Round 8 的经验证明 GT 修正的 ROI 远高于 prompt 微调。

**评估策略调整。** 当前按 `(contentId, category)` 贪心计数匹配。如果某些 FP 实质上是"正确提取但归错了 category"，可以引入部分匹配评分或 category 容错机制——这不是降低标准，而是让评估更准确地反映实际质量。

**L3 泛化探针作为收敛判据。** 当 prompt 在已知数据上稳定满分但剩余 FN 难以归因时，L3 层的对抗探针提供了一种客观的收敛判据：如果 prompt 在所有已知 failure patterns 的措辞变体上仍能正确识别（R18 的 10/10 检出率），则剩余 FN 可归因于 LLM 采样噪声而非系统性缺陷，优化可暂停。反之如果探针暴露出系统性 miss，则明确了下一轮优化的方向。这比"连续 N 轮无变化就停"更有信息量——它区分了"没有改进空间"和"改进空间存在但当前策略不够"。

**稳定性验证作为进入下一阶段的前提。** R19-R23 的 5 轮重复评估是在 prompt 和 GT 都确认收敛后的必要步骤。它的价值不仅是"确认稳定"，更重要的是为后续优化提供精确的显著性判据。没有这个数字，prompt v2.0 的任何改动都无法判断是真实改进还是采样噪声。同时，多轮重复评估还能暴露"系统性弱点"与"随机波动"的区别：R19-R23 中 spec5.impl 的 FP 在 5 轮中均值 5.8、几乎无波动，确认为系统性缺陷；而 R22 的 batch_1 异常是典型的单次采样偏差。这种区分直接决定了 v2.0 的优化方向——攻系统性弱点而非试图"修复"随机波动。

## 七、速查决策表

| 观察到的现象 | 根因判断 | 推荐策略 | 预期效果 |
|-------------|---------|---------|---------|
| 某 category recall 持续极低 | 模型不知道该识别此模式 | 补充边界正例（3-5 个） | recall 大幅提升 |
| 某 category FP 过高 | 模型不知道边界在哪里 | 列出"什么不是 X"的反例 | precision 大幅提升，recall 不变 |
| TP 被归错相邻 category | 缺乏类别间区分规则 | 一句话经验法则 | 混淆率归零 |
| 规则为"建议性"时效果不稳定 | 模型有时遵循有时忽略 | 升级为带步骤的强制检查 | recall 稳定提升 |
| 添加"合并/去重"后 recall 暴跌 | 限制性语义被泛化 | 反转为"独立性原则" | recall 恢复 |
| 修 FP 时 TP 同步下降 | FP 与 TP 语义耦合 | 接受 FP 或拆 pass | — |
| 连续 2+ 轮 REJECT | 触及 prompt 天花板 | 转向架构级方案 | — |
| 剩余 FN 难以判断是缺陷还是噪声 | 泛化能力未验证 | 基于 failure patterns 构造对抗探针 | 确认是否为系统性盲区 |
| 单轮指标异常低，不确定是退化还是波动 | 缺乏稳定性基线 | 多轮重复评估 + 根因定位 | 离群点归因于采样偏差，系统性弱点成为 v2.0 输入 |

---

## 八、适用条件与局限

本手册的策略基于以下实验条件：任务为信息提取 + 分类（非生成式创作）；评估为精确匹配（按 contentId + category 计数）；Ground Truth 规模为 28 篇 golden 文档（含 10 synthetic）、81 条标注（GT v3.4，45 PI + 36 KG）；模型为单次调用（非多轮对话）；并行度为 6 个 subagent。对于生成式任务（如文案撰写）、开放域问答、或评估标准为人类偏好评分的场景，部分策略（尤其是精准反例和经验法则）可能需要适配。但“单变量改动”“限制性语义敏感”“维度隔离”等核心原则具有普适性——它们反映的是 LLM 对指令的理解机制，而非特定任务的特性。
