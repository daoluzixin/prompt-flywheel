# Spec 日报 Prompt 优化实验记录

> 创建日期：2026-05-11
> 当前状态：v2.1 Dual-Pass 架构验证通过（30 轮 eval），PI-only + KG-only 独立 pass 合并评估。最优配置：PI F1=0.84, KG F1=0.87（R30），较 v1.9 单 pass 基线（PI 0.75, KG 0.68）分别提升 +0.09/+0.19。GT v3.4（28 golden docs）。

---

## 一、实验背景与目标

本实验通过数据驱动的 GAN 风格迭代，优化 `analyze-batch-prompt.md`（以下称 G）在 Spec 流程质量分析任务上的表现。G 的职责是分析 Spec 对话文档，提取 `process_issues`（流程不足，简称 PI）和 `knowledge_gaps`（知识库缺口，简称 KG）两类指标。

优化框架遵循 `PROMPT_OPTIMIZATION_AGENT.md` 定义的协议：G 是被优化的 prompt（Generator），评估系统扮演 Discriminator，通过冻结的 Ground Truth 检测 bad case 并反向驱动迭代。核心评估方式是按 `(contentId, category)` 做贪心计数匹配，计算 Precision、Recall、F1。

---

## 二、实验基础设施

目录结构以 `daily-report-optimized/` 为根。`scripts/analyze-batch-prompt.md` 是 G 本体；`eval/` 下包含本实验记录、飞轮协议定义、changelog、metrics.json（全部迭代量化指标）、baseline 目录（V3 冻结 GT 及 44 篇原始文档）、dataset 目录（bad case 判定标准 14 条、失败模式签名库 10 条、hard case 池、对抗生成用例）、scripts 目录（eval-runner.py 及分析脚本）、runs 目录（全部运行结果）。

Ground Truth 规格：V3 schema，当前为 v3.4 版本。GT 经历了 v3.0→v3.1→v3.2→v3.3→v3.4 的演进：v3.1 从 FP 分析中补入 3 条遗漏（2 PI + 1 KG）；v3.2 新增 pi-035（overview 描述错误）；v3.3 新增 pi-036（文件命名规则纠错）；v3.4（R18 起）新增 10 篇 synthetic golden 文档（来自 L3 对抗探针）。当前 GT v3.4 总计 28 golden（18 original + 10 synthetic）+ 26 silver = 54 篇，45 PI + 36 KG。其中原始 18 篇含 30 PI（easy 14 / medium 7 / hard 9）+ 22 KG（easy 7 / medium 11 / hard 4）；PI 按阶段分布为 spec1.spy 8、spec2.clarify 1、spec3.plan 12、spec4.tasks 2、spec5.impl 7；KG 按类型分布为系统架构 11、业务规则 4、技术栈 1、接口约束 1、数据模型 5。

评估流程：由 6 个 subagent 并行执行 G 输出 JSON，eval-runner.py 汇总后按 `(contentId, category)` 匹配 GT 计算 TP/FP/FN，再按 category 和 difficulty 输出细分指标，与上一版本自动对比标记退化项。R1-R17 在 18 篇原始 golden 文档上评估（6 batch × 3 篇）；R18 起扩展至 28 golden 文档（6 batch × ~5 篇），包含 10 篇 L3 合成探针。

Batch 分配：batch_0（1c1bae3d, 3bbf3b4d, 78e884b3）、batch_1（63b4d6c5, 8581e6a8, 94f4fa6b）、batch_2（0d025ed0, 71ab5b4c, 04d6bbe9）、batch_3（1ef84a84, fadb2be3, cda1bacd）、batch_4（b0419fc0, 187cf841, 734b79d3）、batch_5（f39aa58c, be99243f, 2757704387）。

## 三、Baseline 建立（Run 1-4）

使用同一 G v1.0（hash `239e849a5d4f`）连续运行 4 轮建立方差基线。PI F1 四轮分别为 0.67 / 0.52 / 0.65 / 0.60，均值 0.61（标准差 0.06）；KG F1 分别为 0.73 / 0.43 / 0.65 / 0.65，均值 0.61（标准差 0.12）。KG 波动显著大于 PI，说明 KG 判定更受随机性影响。

Category 维度的关键发现：spec1.spy 平均 recall 仅 0.21，是系统性漏检的最差类别；spec2.clarify 四轮全漏（recall 0.00）；系统架构 FP 波动极大（2~9）。难度维度上，easy PI recall 为 0.86，medium 为 0.68，hard 仅 0.18——Run2 甚至为 0.00，这是最严重的短板。

四条核心结论：spec1.spy 系统性漏检（FP_001，severity 3）是最优先修复目标；Hard PI 整体低召回（FP_002）；KG 跨轮波动大（FP_006）；系统架构过度生成（FP_003）。

## 四、Root Cause 分析

spec1.spy 漏检的根因在 `be99243f-catpaw`（贡献 6 条 spec1.spy PI）。该文档的 spec1.spy 信号全部来自用户对 spec.md 的 spec-mark 短批注（2-5 字，如"MQ消息""支持可配置""这些不全"），5/6 条标记为 hard 难度。G v1.0 的 spec1.spy 正例仅有标准长度的纠错句，完全没有关于 spec-mark 批注作为 PI 信号的说明。模型不是"不愿识别"而是"不知道应该识别"——这是 prompt 知识缺口，非模型能力限制。

## 五、飞轮迭代详细记录

### Round 1（v1.0 → v1.1, hash ba68622ddc4f）

目标是 FP_001（spec1.spy 漏检）。在 G 中新增"spec1.spy 阶段的纠错"段落（lines 74-82），描述 spec-mark 批注作为 PI 信号的判定规则，并在正例区补充 3 个 spec1.spy 示例。这些示例刻意选用了极短的批注（"MQ消息""支持可配置""这些不全"），教会模型"短 ≠ 不重要"的判断。

Run5 结果：spec1.spy recall 从 0.21 飙升至 1.00，PI F1 从 baseline 0.61 升至 0.71，KG F1 升至 0.72，Hard PI recall 从 0.18 跃至 0.86。Precision 下降 3-6pp 是 recall 提升的正常代价，FP 主要集中在系统架构 KG（7 个）。决策 KEEP。

### Round 2（v1.1 → v1.2, hash 2c5d5ebc678e）

目标是 FP_003（系统架构 KG 过度生成，Run5 中 7 个 FP）。逐条分析 FP 后识别出 5 类典型误判模式，根因是 G 缺乏"什么不是系统架构"的负向约束。新增数据模型边界区分规则和"不属于系统架构的内容"反例段落（spec 流程规范、模板设计说明、框架愿景、方法论讨论、知识库分类体系）。

Run6 结果：系统架构 FP 从 7 暴降到 1，KG Precision 从 0.62 飙升至 0.90（+45%），同时 Recall 不降反升。PI F1 无退化（-0.01）。cda1bacd 文档的 3 个概念性 FP 全部被正确过滤。决策 KEEP。

### Round 3（v1.2 → v1.3, hash 06cbf186415e）

目标是 spec4.tasks category 混淆（Run6 中 recall 为 0.00，GT 2 条均被误归为 spec3.plan 或 spec5.impl）。根因是 G 对 spec4 的描述不足以区分相邻阶段——"恢复删掉的步骤"和"按维度筛选"被误判为 plan 层面问题。新增 spec4.tasks 专属判定段落，给出 3 类场景描述和一条经验法则："what to do（做什么）层面的问题→spec4；how to do（如何设计/实现）→spec3/spec5"，并补充 2 个正例。

Run7 结果：spec4.tasks 从 recall 0.00 恢复到 1.00，PI F1 升至 0.80（Precision 0.90 为历史最高），spec1.spy 维持完美 F1=1.00。KG F1 下降 0.05 在 baseline 方差范围内。综合 F1 从 0.80 升至 0.825。决策 KEEP。

### Round 4（v1.3 → v1.4, hash 0c653227bc39）

目标是 spec3.plan 漏检（Run7 中 FN=4，最大 PI 失败类别）。分析发现 4 条 FN 涵盖"维度选择错误被误归 spec4""完全漏报的拆分逻辑建议""改进建议型未被识别"等模式，根因是 G 对 spec3.plan 无专属描述段。新增约 25 行内容：5 个场景描述 + 识别要点 + 与 spec4 的边界规则 + 3 个正例 + "补充信息"与"批评设计思路"的区分注释。

Run8 结果：spec3.plan Recall 从 40% 升至 70%（+30pp），PI Recall 从 0.72 升至 0.80，Hard PI recall 从 0.57 升至 0.71。代价是引入 3 个 spec3.plan FP（Precision 从 0.90 降至 0.80）。KG 几乎无影响。决策 KEEP。

### Round 5（v1.4 → v1.5b, hash 1178efc7fd59）

目标是 PI Precision 回收。Round 5a 添加了 spec-mark "合并原则"导致 PI Recall 从 0.80 暴跌至 0.68，spec1.spy recall 从 1.00 降至 0.67——教训深刻：对 LLM 说"合并"会严重抑制输出倾向，即使限定了"只限完全相同"。Round 5b 将合并原则反转为"独立性原则"（每条独立事实性错误单独计 PI），同时保留 spec3.plan 反例（知识文件存放位置建议→KG 非 PI、间接提到 plan 但目的是补充背景→不计入）。

Run9b 结果：PI F1 首次突破 0.84 创五轮新高，spec3.plan F1 从 0.70 升至 0.86，Easy+Medium PI 达到完美 100% recall，所有剩余 FN 集中在 Hard。KG 业务规则 FP=4 成为新瓶颈。决策 KEEP v1.5b。

### Round 6（v1.5b → v1.6, hash 78f3a25a28f4）

目标是 KG Precision 回收（业务规则 FP=4 + 系统架构 FP=2，共 FP=6）。在 KG"不计入条件"中新增 2 条排除规则：「Spec 流程/模板的设计理念和方法论」不属于可沉淀的业务背景知识；「AI/框架的概念性定位或愿景描述」不属于稳定背景事实。

Run10 结果：KG Precision 从 0.76 升至 0.857（+9.7pp），业务规则 F1 从 0.50 跃至 0.80。PI F1 退化 3pp（0.846→0.816），推测为子 agent 随机性波动（spec5.impl 和 spec3.plan 各有 2 个 hard FN）。综合 F1 持平于 0.837。决策 KEEP。

### Round 7（v1.6 → v1.7, hash 6887e5143806）— REJECTED

目标是恢复 PI Recall（cda1bacd 的 spec3.plan FN=2 疑似被 R6 KG 排除规则间接抑制）。尝试在 KG 排除规则末尾加注"此排除仅限 KG 维度，PI 不受影响"。

Run11 结果全面退化：PI Precision 从 0.83 暴跌至 0.75（FP 4→7），KG Precision 从 0.857 降至 0.783（业务规则 FP 2→5），综合 F1 从 0.837 退化到 0.805。教训：在排除规则旁添加"另一维度不受影响"的元声明，被模型理解为鼓励多输出的信号。回滚到 v1.6。

## 六、总成绩与演进轨迹

从 baseline 到当前最优配置（G v1.9 + GT v3.3），PI F1 从 0.61 提升到 0.93（+0.32），KG F1 从 0.61 提升到 0.95（+0.34），综合 F1 从 0.61 升至 0.94。优化分两个阶段：第一阶段（R1-R11，prompt-only）通过 11 轮 prompt 迭代将 F1 从 0.61 推至 0.91，累计约 60 行 prompt 增量改动；第二阶段（R12-R17，数据飞轮闭环）同时优化 prompt（v1.8→v1.9，加入 spec2.clarify 指引和 KG 边界细化）和 GT（v3.0→v3.3，从 FP 分析中补入 5 条遗漏），将 F1 进一步推至 0.94，并通过 3 轮连续评估验证了稳定性。

Category 攻克情况：spec1.spy 从 recall 0.21 到 1.00（R1 完全解决）；spec4.tasks 从 0.50 到 1.00（R3 完全解决）；spec5.impl 从 FN=2 到完美 F1=1.00（R8 解决）；系统架构 FP 从均值 4 降至 0（R2+R6+飞轮最终解决）；spec3.plan 从 0.40 提升到 F1 0.91（R4+R5+R9+飞轮逐步解决）；业务规则从 F1 0.50 到 0.89（R9+飞轮解决）；spec2.clarify 从 11 轮 recall 0.00 到飞轮阶段 F1=1.00（prompt v1.9 的 spec2.clarify 指引彻底攻克，连续 3 轮稳定）。

## 七、已知瓶颈与剩余问题

经过飞轮闭环后，此前的多个瓶颈已被突破。spec1.spy FP=3 在第一阶段（R10/R11）被证明是 prompt 层面的天花板，但飞轮阶段通过 GT 补完（确认部分 FP 实为 GT 遗漏）和 prompt v1.9 的组合改动，FP 在 R15-R17 中降至 0-1 的范围，不再是主要问题。spec2.clarify 在第一阶段 11 轮 recall=0.00，在 prompt v1.9 添加专门的 spec2.clarify 指引后，连续 3 轮（R15/R16/R17）稳定达到 F1=1.00，彻底攻克。

当前剩余问题：PI 侧有 3 个 FN（spec1.spy 1 + spec3.plan 2），均为 hard 难度边界 case；KG 侧有 1 个 FN（接口约束 category，easy 难度但 R17 仍 miss）。这些 FN 的稳定性需要进一步观察——如果连续多轮都是同一条 miss，可能是 prompt 的系统性盲区；如果是随机波动，则属于 LLM 采样噪声的正常范围。当前综合 F1=0.94 已非常接近理论天花板。

## 八、每轮复盘：核心改动、改进与不足

Round 1 的核心改动是为 spec1.spy 补充 spec-mark 短批注的模式识别教学。改进：一举解决了最严重的系统性漏检（recall 0.21→1.00），同时带动 Hard PI recall 从 0.18 跃至 0.86。不足：只顾攻 recall 未设防 precision，系统架构 KG 7 个 FP 被放任，为下一轮埋下了负担。

Round 2 的核心改动是为系统架构 KG 添加 5 类"什么不是系统架构"的负向反例。改进：FP 从 7 降到 1，KG Precision 从 0.62 一步跳到 0.90，且 Recall 不降反升——证明精准反例的杠杆效应极大。不足：没有同步巩固 PI recall，spec4.tasks 在这一轮暴露为 0 recall（虽非本轮引入，但属于被忽视的盲区）。

Round 3 的核心改动是为 spec4.tasks 建立独立的判定段落和"what to do vs how to do"经验法则。改进：spec4 从完全失败恢复到 recall 1.00，PI Precision 同时达到历史最高 0.90。不足：spec3.plan recall 停滞在 0.60（FN=4）暴露为最大瓶颈，说明此前对"plan 阶段纠错"的教学完全缺失。

Round 4 的核心改动是为 spec3.plan 新增专属描述段（5 个场景 + 3 个正例 + 与 spec4 边界规则）。改进：spec3.plan recall 从 40% 升至 70%（+30pp），Hard PI recall 升至 0.71。不足：引入 3 个新 FP（Precision 从 0.90 降至 0.80），说明正例教学力度偏强、缺乏配套的反例约束来控制边界。

Round 5 的核心改动经历了一次失败再修正。5a 添加"合并原则"导致 recall 暴跌——教训是对 LLM 说"合并"等于全面抑制输出。5b 反转为"独立性原则"并补充 spec3.plan 反例。改进：PI F1 首次破 0.84，Easy+Medium 达到 100% recall。不足：KG 业务规则 FP=4 未被触及，成为新的精度瓶颈；且 5a 的失败浪费了一次完整 eval 运行。

Round 6 的核心改动是在 KG 不计入条件中新增 2 条排除规则（spec 模板方法论 + AI 概念性定位）。改进：KG Precision 从 0.76 升至 0.857，业务规则 F1 从 0.50 跃至 0.80。不足：PI F1 退化 3pp（可能为随机波动但无法完全排除规则间接影响），且未做 stability check 验证。

Round 7 的核心改动是在 KG 排除规则旁添加"此排除仅限 KG 维度"的元声明。改进：无——全面退化。不足：这是一次典型的"过度解释"错误，元声明被模型理解为鼓励多输出的隐含信号，PI FP +3、KG FP +3，综合 F1 跌 3pp。教训是排除规则应当简洁自足，不应附加跨维度声明。

Round 8 的核心改动包括两部分：(1) 新增"spec5.impl 与 spec1.spy 的易混淆场景"段落，明确 spec5 执行阶段遗漏服务归 spec5.impl 而非 spec1.spy；新增"脚本设计改进建议也是 spec5.impl"的示例。(2) 修正 GT 标注错误——pi-015/pi-016 的 quote 物理存在于 fadb2be3 而非 cda1bacd，将其归属修正。(3) 添加"一份文档可以同时包含 PI 和 KG"的提示。改进：PI F1 从 0.79 跃至 0.86（+7pp），spec5.impl 从 FN=2 达到完美（F1=1.00），spec3.plan FP 从 3 降到 1。不足：KG Recall 从 0.86 降到 0.67（-19pp），模型变得过于保守漏掉了 7 个 KG（主要是系统架构 FN=5）。KG Precision 升至 1.00 但代价是 recall 大幅下降。整体 PI 改善显著，KEEP。

Round 9 的核心改动聚焦 KG Recall 恢复：(1) 将纠错伴生知识规则从"建议"升级为"强制执行的二次检查"——要求模型在输出 JSON 前逐条回顾每条 PI，检查是否包含文件命名规则/维度变更/通信方式等伴生知识点，并给出具体伴生模式示例。(2) 在 KG 提取完整性要求中新增"通信方式与消息内容是独立知识点"的拆分指导。(3) 在 spec 方法论排除规则后新增"注意区分"提示——即使文档主要讨论 spec 设计，其中穿插的系统实际机制纠正仍应输出 KG。改进：KG F1 从 0.80 跃至 0.93（+13pp），KG Recall 从 0.67 飙至 0.90（+24pp），系统架构从 F1 0.67 升至 0.90，业务规则达到 F1 1.00。PI F1 同步提升至 0.88（+2pp），spec3.plan 达到 F1 0.90，Easy/Medium 全面 100% recall。不足：spec1.spy FP=3 未解决；Hard KG 仍有 2 个 FN（可能为 GT 边界争议项）。KEEP。

Round 10 的核心改动是在 spec1.spy 段落中增加 PI/KG 边界判断规则和"计数自检"步骤，要求模型在输出 spec1.spy 前确认"spec.md 哪里写错了"。改进：PI F1 跃至 0.96（spec1.spy 达到完美 F1=1.00，FP 从 3 降到 0），spec3.plan recall 也达 1.00。不足：KG F1 从 0.93 暴跌至 0.83（-10pp），系统架构 FP 从 1 增到 3，业务规则和接口约束各新增 1 个 FN。教训：添加跨维度（PI/KG）的宏观边界规则会干扰 KG 提取行为——模型把"这不是 PI"理解为"这可能也不该输出为 KG"。REJECT。

Round 11 换用"精确反例"策略：在反例区增加 3 个具体的 spec1.spy FP 场景（redo重试、MQ消息体字段、重复标注），不添加宏观规则。改进：spec1.spy FP 确实降到 0。不足：spec1.spy TP 从 6 暴跌至 3（recall 0.50），模型看到反例后过度保守，把有效的 spec1.spy PI 也吞掉了。系统架构 FP 反而增到 4。这再次验证了 Round 5a 的教训——任何涉及"不要输出 X"的措辞（即使是精确反例形式）都会让 LLM 全局性地抑制相关输出。REJECT。

总体来看，第一阶段 11 轮中 7 轮 KEEP、4 轮 REJECT。KEEP 的轮次共性是"单一目标 + 精准教学/约束"；REJECT 的轮次共性分三类：(a) 间接措辞（合并原则、元声明），(b) 跨维度宏观规则（PI/KG 边界），(c) 涉及"不要输出"的反例（即使精确到具体 quote 也会抑制有效输出）。R10-R11 碰到天花板后，实验转入第二阶段：数据飞轮闭环。

### 第二阶段：数据飞轮闭环（R12-R17）

飞轮的核心思路是"不再只改 prompt，同时改数据"——从 FP 分析中发现 GT 遗漏，补入后暴露新的真实 FN，再针对性改 prompt，形成 prompt-data 双向驱动的闭环。

**Prompt v1.8 → v1.9 的改动**（R12-R14 期间完成）：
1. 新增 spec2.clarify 专属指引——描述 spec2 产出的 outline.yml 中路由条目被用户删除/spec-mark 标记错误时，应识别为 spec2.clarify 类型的 PI。给出 1 个正例（"校园订单筛选与企客权限隔离查询能力"被用户通过 spec-mark 要求删除）。这一改动直接攻克了 11 轮未解的 spec2.clarify recall=0.00 问题。
2. KG 边界细化——在"Spec 模板扩展位"相关内容旁添加边界说明：当用户纠正的是扩展位的实际结构定义（如 extensions 字段的具体含义），仍应输出为 KG；只有讨论"模板为什么设计扩展位"的方法论部分才排除。

**GT v3.0 → v3.3 的演进**（从 R15/R16 的 FP 分析中发现并补入）：
- v3.1：补入 3 条 GT 遗漏——pi-009b（be99243f，spec1.spy，渲染建议实为 spec.md 纠错）、pi-034（be99243f，spec1.spy，MQ 消息体字段纠错）、kg-022（1c1bae3d，系统架构，按能力维度筛选服务规范）。
- v3.2：补入 pi-035（1c1bae3d，spec4.tasks，overview 描述与实际不符）。
- v3.3：补入 pi-036（1ef84a84，spec3.plan，文件命名规则纠错）。

**R15（Run 19）— prompt v1.9 首次评估**：对 GT v3.2 跑出 PI F1=0.8772, KG F1=0.8571。spec2.clarify 首次达到 recall=1.00（F1=1.00）。Hard PI recall=0.89。

**R16（Run 20）— 稳定性验证**：第二次跑 prompt v1.9。batch_1.json 出现 JSON 解析错误（未转义的中文引号），修复后重跑。对 GT v3.2 的结果与 R15 高度一致。随后用 GT v3.3 重新评估 R15/R16 的预测结果，PI F1 均提升至 0.93（GT 补完消除了 FP），验证了飞轮"GT 补完 → 指标自然提升"的机制。

**R17（Run 21）— 飞轮闭环验证**：直接对 GT v3.3 跑新一轮预测。结果：PI F1=0.9310（P=0.9643, R=0.9000），KG F1=0.9545（P=0.9545, R=0.9545）。所有难度层级 PI recall=1.00；KG 仅 easy 层有 1 条 miss（R=0.86），medium/hard 全部 100%。spec2.clarify 连续第 3 轮 F1=1.00，确认已稳定攻克。系统架构 category 达到完美（TP=11, FP=0, FN=0）。

三轮（R15/R16/R17）的一致性验证了飞轮闭环的有效性：prompt 改进（v1.9 解决 spec2.clarify）+ GT 补完（v3.3 消除假 FP）的组合将综合 F1 从 0.91 稳定推至 0.94，且波动极小。

## 九、方法论沉淀

> 完整策略手册见 `BEST_PRACTICES.md`，本节为核心结论的浓缩。

十一轮实验揭示了一条根本性的规律：LLM 对限制性指令的敏感度远超预期。所有 7 次 KEEP 的共性是"正向教学"——教模型什么是、什么属于、该怎么做；所有 4 次 REJECT 的共性是"负向约束"——告诉模型什么不是、什么不该、要合并要减少。这不是巧合，而是 LLM 理解指令的底层机制决定的：限制性措辞（"合并""不要输出""此规则仅限 X"）会被泛化为全局性的保守策略，而正向描述（"每条独立计数""输出前逐条回顾"）则被精确执行。

七条经验法则按 ROI 排序：单一目标 + 单变量改动（归因基础，无例外）；正例教学解决系统性漏检（recall 从 0.21→1.00，效果最大的单次操作）；精准反例解决过度生成（Precision +45%，但对高 recall category 有天花板）；一句话经验法则解决 category 混淆（比三段定义有效）；"建议性"升级为"带步骤的强制检查"（KG Recall +24pp）；独立性原则替代合并原则（避免 recall 暴跌）；维度间规则严格隔离（任何跨维度引用都会双向退化）。

四类失败模式各有天花板。系统性漏检几乎无天花板，正例选得准就能解决；过度生成在 FP 与 TP 语义耦合时触顶（Round 11：FP 归零但 TP 从 6 降到 3）；Category 混淆在相邻阶段的 hard case 处触顶；跨维度语义耦合在 prompt 层面完全无解——Round 10 和 Round 11 分别用宏观规则和精确反例两条完全不同的路径尝试，都导致非目标维度连带退化。最后一类问题的解法只能是架构级的：将多维度提取拆为独立 pass，或引入二阶段"先提取后分类"pipeline。

工程层面的关键纪律：先建立 baseline 方差（本实验 KG σ=0.12，小于此值的变化不可信）；每次 KEEP 后立即备份（4 次 REJECT 靠此零成本回滚）；怀疑 prompt 到极限时先查 GT 标注质量（Round 8 修正 GT 错误后单轮涨 7pp，比任何 prompt 改动都大）。

## 十一、L3 合成泛化探针首次触发（R18）

> 时间：2026-05-11  目标：验证 prompt v1.9 在对抗性合成样本上的泛化能力

### 11.1 L3 层设计

L3 是数据飞轮的第三层——"D 切换为出题者角色"。核心思想：将飞轮过程中暴露的 failure patterns 沉淀为 `failure_patterns.jsonl`（13 条，10 resolved + 3 active），当 active 模式积累到一定量后，基于这些模式的措辞变体和场景变体定向生成对抗样本（adversarial probes），用于检测 prompt 是否仅在已知 case 上过拟合、或具有真正的泛化能力。

### 11.2 Active Failure Patterns 来源

3 个 active 模式均来自 R17 FN 分析：

- **FP_011**（短文档零输出）：734b79d3 仅 2 轮对话讨论接口鉴权，模型输出零结果。根因：文档太短 + 非 spec 氛围 → 模型触发"短文档/无技术讨论"的 bail-out 逻辑
- **FP_012**（极短泛化批注遗漏）：be99243f 中 pi-020"这些不全"（仅 4 字）被漏检。根因：虽然 R1 已解决 spec-mark 短批注，但"这些不全"不是 spec-mark 而是对话中的泛化性短否定
- **FP_013**（提问式纠错遗漏）：1ef84a84 中 pi-003"你能不能做到…"被漏检。根因：提问句式（能不能/可不可以/是不是应该）被理解为"征询建议"而非"纠错"

### 11.3 合成探针设计

生成 10 个 probes 写入 `dataset/synthetic/probes.json`，每个 probe 包含模拟对话片段 + 期望输出 + 设计意图。覆盖：

- FP_013 × 3 probes："能不能简洁点"、"可不可以换种方式"、"是不是应该拆开"。均为提问句式的 PI

Probes 被补入 GT v3.4（新增 10 synthetic golden docs），GT 总量扩展至 54 篇（28 golden + 26 silver），36 KG + 45 PI。

## 12. Prompt v1.9 稳定性验证（R19-R23）

### 12.1 实验目的

L3 合成探针验证了 prompt v1.9 的泛化能力，但单次评估（R18）无法量化 subagent 随机波动对指标的影响。本轮连续 5 次评估（R19-R23）在完全相同条件下（prompt v1.9 不变、GT v3.4 冻结、28 golden docs）重复运行，目的是建立稳定性置信区间，区分系统性缺陷与随机噪声。

### 12.2 实验配置

所有 5 轮参数完全一致：prompt v1.9（417 行，hash 不变）、GT v3.4（28 golden = 18 original + 10 synthetic）、6 并行 subagent 每批 9 篇、greedy count matching 评估。唯一变量是 LLM 采样随机性。

### 12.3 汇总结果

| Round | PI P | PI R | PI F1 | KG P | KG R | KG F1 |
|-------|------|------|-------|------|------|-------|
| R19 | 0.75 | 0.75 | 0.75 | 0.62 | 0.75 | 0.68 |
| R20 | 0.76 | 0.69 | 0.72 | 0.63 | 0.71 | 0.67 |
| R21 | 0.76 | 0.89 | 0.82 | 0.67 | 0.79 | 0.72 |
| R22 | 0.62 | 0.64 | 0.63 | 0.50 | 0.61 | 0.55 |
| R23 | 0.76 | 0.86 | 0.81 | 0.72 | 0.82 | 0.77 |
| **Mean** | **0.73** | **0.77** | **0.75** | **0.63** | **0.74** | **0.68** |
| **Std** | **0.06** | **0.10** | **0.08** | **0.08** | **0.08** | **0.08** |

5 轮均值：PI F1=0.75（σ=0.08），KG F1=0.68（σ=0.08）。

### 12.4 R22 异常分析

R22 是明显离群点（PI F1=0.63，低于均值 1.5σ）。根因定位于 batch_1 的 spec1.spy 维度：该批次产生 6 个 FP（precision=0.25），同时漏检 8 个 FN。对比同批次在其他 4 轮的表现（R19/R20/R21/R23 的 spec1.spy FP 分别为 0/1/1/1），R22 的 spec1.spy 异常是典型的 subagent 单次采样偏差——该 subagent 在本轮将大量对话片段误判为"需求初始阶段的 PI"，导致 FP 激增。

进一步佐证：R22 的 hard 难度 PI recall 仅 0.42（对比 R21 的 1.0 和 R23 的未缺失），KG hard recall 降至 0.0（对比 R21 的 0.5），说明该轮 subagent 在复杂判断场景集体表现不佳。R23 立即恢复至 PI F1=0.81、KG F1=0.77 的高位，进一步确认这是随机波动而非 prompt 退化。

### 12.5 排除离群点后的稳定带

剔除 R22 后（R19/R20/R21/R23 四轮）：

| Metric | Mean | Std | Range |
|--------|------|-----|-------|
| PI F1 | 0.78 | 0.05 | [0.72, 0.82] |
| KG F1 | 0.71 | 0.05 | [0.67, 0.77] |

四轮标准差收窄至 0.05，说明 prompt v1.9 在正常采样条件下具有良好的确定性。PI F1 的 95% 置信区间约为 [0.70, 0.86]，KG F1 约为 [0.63, 0.79]。

### 12.6 关键发现

**Precision 稳定性优于 Recall**：PI precision 在 5 轮中维持 0.62-0.76（排除 R22 后为 0.75-0.76，几乎无波动），而 recall 的波动范围更大（0.64-0.89）。这说明 prompt v1.9 的"什么不该提取"规则已经收敛，但"什么必须提取"仍受 subagent 注意力随机性影响。

**spec5.impl 持续高 FP**：5 轮中 spec5.impl 的 FP 数为 7/5/7/3/7，均值 5.8，远高于其他 PI category。这是当前 prompt 的系统性弱点——subagent 倾向于将"代码层面的实现讨论"过度标记为 PI，即使对话中只是在解释实现细节而非提出改进建议。这一发现可作为 prompt v2.0 的优化方向。

**KG 的"接口约束"category 噪声大**：5 轮中接口约束的 FP 数为 4/5/3/4/（R23 待确认），均值约 4，与业务规则的 FP 相当。两者的语义边界模糊是已知问题（syn-010 已暴露），但 5 轮重复验证确认这是一个稳定的系统性 FP 来源，而非随机波动。

### 12.7 结论

5 轮稳定性验证确认 prompt v1.9 在 28-doc golden 基准上的真实性能为 **PI F1≈0.75-0.78、KG F1≈0.68-0.71**（取决于是否剔除离群点）。单轮评估的随机波动幅度约 ±0.08（1σ），极端情况（如 R22）可达 -0.12。这一波动水平意味着：后续 prompt 优化的增益必须超过 0.10 F1 才能被可靠地判定为真实改进而非采样噪声。

prompt v1.9 的两个系统性弱点已清晰定位：spec5.impl 的 FP 倾向和接口约束/业务规则的 category 边界模糊。这两者将作为 prompt v2.0 迭代的优先优化目标。

## 十三、架构级实验：独立 Pass 拆分（R26）

### 13.1 实验假设

将 PI 和 KG 提取拆为两个物理独立的 pass（PI-only + KG-only），从根本上消除跨维度语义耦合，解决：
1. spec5.impl 系统性 FP（均值 5.8）：单 pass 中 KG 规则的存在让模型对"代码相关内容"产生注意力偏置
2. 接口约束/业务规则 category 模糊：单 pass 认知负荷导致 KG 细粒度分类边界模糊

### 13.2 实验设计

- PI-only prompt（`scripts/analyze-batch-prompt-pi-only.md`）：仅保留 PI 相关规则，新增 spec5.impl 严格边界"三条件同时满足"判定规则
- KG-only prompt（`scripts/analyze-batch-prompt-kg-only.md`）：仅保留 KG 相关规则，移除纠错伴生知识检查，展开 category 判定指引
- 合并脚本（`eval/scripts/merge-dual-pass.py`）：按 batch_index 对齐合并两个 pass 的输出
- 每个 pass 各 6 个 subagent 并行执行，合并后用相同的 eval-runner.py 评估

### 13.3 结果

| Metric | 单 Pass 基线（R19-R23 Mean） | 独立 Pass (R26) | Delta |
|--------|-----|------|-------|
| PI Precision | 0.73 | **0.8205** | +0.09 |
| PI Recall | 0.77 | **0.8889** | +0.12 |
| PI F1 | 0.75 | **0.8533** | **+0.10** |
| KG Precision | 0.63 | **0.7037** | +0.07 |
| KG Recall | 0.74 | 0.6786 | -0.06 |
| KG F1 | 0.68 | **0.6909** | +0.01 |

### 13.4 Category 细分分析

**PI 侧（核心改进）**：

| Category | 单 Pass 基线 FP（均值） | 独立 Pass FP | 变化 |
|----------|---|---|---|
| spec5.impl | 5.8 | **1** | **-4.8（降幅 83%）** |
| spec3.plan | ~2 | 4 | +2（略升） |
| spec4.tasks | ~1 | 2 | +1（略升） |
| spec1.spy | ~1 | 0 | -1 |
| spec2.clarify | 0 | 0 | 不变 |

spec5.impl FP 从均值 5.8 骤降至 1，验证了"三条件同时满足"规则 + 物理隔离的有效性。

**KG 侧**：

| Category | TP | FP | FN | F1 |
|----------|---|---|---|---|
| 系统架构 | 7 | 1 | 6 | 0.67 |
| 业务规则 | 4 | 2 | 1 | 0.73 |
| 技术栈 | 2 | 0 | 0 | 1.00 |
| 接口约束 | 2 | 3 | 1 | 0.50 |
| 数据模型 | 4 | 2 | 1 | 0.73 |

接口约束 FP=3 仍然偏高（基线约 4），category 边界模糊问题在独立 pass 中并未完全解决——可能需要更细的判定规则而非仅靠物理隔离。系统架构 recall 偏低（0.54），FN=6 是当前最大失分项。

### 13.5 关键发现

1. **PI F1 提升 +0.10 超过 significance threshold（0.08）**，属于真实改进而非采样噪声
2. **spec5.impl FP 从 5.8 降至 1**，是本实验最大的胜利——"三条件同时满足"判定规则+物理隔离双重保障有效抑制了过度标记
3. **KG 侧改进有限**（F1 仅 +0.01），系统架构 recall 下降 + 接口约束 FP 未解决。KG 的瓶颈不在于跨维度耦合，而在于 category 内部判定指引不够细致
4. **移除"纠错伴生知识"规则的代价**：KG recall 从 0.74 微降至 0.68，与预期一致——原先由 PI 侧带出的伴生 KG 在独立 pass 中需要 KG-only pass 自行发现，存在少量遗漏
5. **PI difficulty recall 优异**：easy 0.93, medium 1.00, hard 1.00——独立 pass 的专注度让模型在 hard case 上也不再遗漏

### 13.6 决策

**PI-only pass: KEEP（显著改进）**。PI F1 从 0.75 提升至 0.85，spec5.impl FP 从 5.8 降至 1，远超预定的成功标准。

**KG-only pass: MARGINAL**。KG F1 仅从 0.68 微升至 0.69，未达到 +0.10 的显著性门槛。但未出现退化（-0.01 在噪声范围内）。

**综合决策：KEEP 独立 Pass 架构作为 v2.0 基础**，后续在 KG-only pass 上单独迭代以解决系统架构 recall 和接口约束 FP 问题。

### 13.7 后续方向

1. **KG-only prompt 迭代**：补充系统架构的正例教学（当前 FN=6 是最大失分项），细化接口约束 vs 业务规则的经验法则
2. **恢复伴生知识机制的变体**：在 KG-only pass 中引入"检查用户纠正 AI 时补充的事实"的规则（不引用 PI 概念），恢复 recall
3. **PI-only prompt 微调**：spec3.plan FP=4 仍可优化，可参考 R2 的精准反例策略

## 十五、R29-R30 KG-Only Prompt 针对性优化

### 15.1 背景

基于十四节稳定性测试的结论，KG-only prompt 存在两个主要问题：(1) 系统架构 FN=5-6（Recall 低）；(2) 业务规则/数据模型 FP 从负例文档大量生成。R29-R30 为 KG-only prompt 的 GAN 式迭代，PI pass 冻结复用 R28 结果。

### 15.2 R29：FP 消除 + 系统架构 Recall 提升

**诊断**：通过对比 R28 预测与 GT，定位到精确原因：
- 系统架构 FN=5：kg-001b（按能力维度筛选）、kg-005/006（服务维度组织）、kg-007（plan字段填写边界）、kg-008（本地收集机制）
- 业务规则 FP=6-8：主要来自 `8581e6a8`（负例文档，GT KG=0）和 `b0419fc0`（GT KG=0），模型将 PRD 需求文本中的业务参数误判为知识库缺口
- 数据模型 FP=3：同上两文档的枚举值/字段路径被错误提取

**Prompt 变更（单变量）**：
1. 新增不计入条件："需求分析/方案设计中的业务逻辑阐述" — 区分 PRD 原文 vs 纠正性补充
2. 新增系统架构正例 5 条：文件组织规则、收集维度调整、能力维度筛选、本地收集机制、模块间职责边界
3. 新增反例 3 条：直接针对企客校园送场景的 FP pattern

**R29 结果**：

| 指标 | R28 | R29 | Delta |
|------|-----|-----|-------|
| KG Precision | 0.6452 | 0.8276 | **+0.1824** |
| KG Recall | 0.7143 | 0.8571 | **+0.1428** |
| KG F1 | 0.6780 | **0.8421** | **+0.1641** |
| 系统架构 FN | 5 | 0 | -5 |
| 业务规则 FP | 4-7 | 0 | -4~7 |
| 数据模型 FP | 3-5 | 0 | -3~5 |

关键突破：系统架构 Recall 从 ~60% 跃至 **100%**，业务规则 Precision=1.00。

**判定：KEEP。** F1 提升 +0.16 远超显著性阈值 σ=0.08。

### 15.3 R30：Category 边界精细化

**诊断**：R29 剩余 FP=5, FN=4，主要为 category 误分类：
- `f39aa58c` kg-010（GT: 业务规则）被预测为"接口约束" — 订单号获取优先级逻辑
- `syn-010` kg-syn-010（GT: 业务规则）被预测为"接口约束" — 幂等性约束
- `734b79d3` kg-009（GT: 接口约束）被预测为"系统架构" — 接口无鉴权
- `fadb2be3` "plan模板字段分类列表" 为 FP — 模板结构描述

**Prompt 变更（单变量）**：
1. 新增 category 区分规则 3 条：接口鉴权→接口约束、获取优先级逻辑→业务规则、幂等性约束→业务规则
2. 新增不属于系统架构排除项：模板字段分类列表

**R30 结果**：

| 指标 | R29 | R30 | Delta |
|------|-----|-----|-------|
| KG Precision | 0.8276 | 0.8125 | -0.0151 |
| KG Recall | 0.8571 | **0.9286** | **+0.0715** |
| KG F1 | 0.8421 | **0.8667** | **+0.0246** |
| 业务规则 | F1=0.75 | **F1=1.00** | +0.25 |
| 接口约束 | F1=0.57 | **F1=1.00** | +0.43 |
| 系统架构 FP | 2 | 5 | +3 (新 FP 来自 cda1bacd/187cf841) |

Recall 大幅提升（FN 从 4→2），Precision 微降（系统架构新增 3 个边界 FP），总体 F1 再提 +0.025。

**判定：KEEP。** F1 进一步提升，且 Recall 接近 93%。Precision 微降来自 cda1bacd/187cf841 的边界性 KG 条目，属于可接受的 tradeoff。

### 15.4 R29-R30 综合效果
